# 前端界面模块 (frontend-ui)

> 无框架纯浏览器端仪表盘，三面板暗色布局，通过 HTTP 调用后端 API，实现 Prompt 压缩全流程交互。
>
> **修订: 2026-07-07** — 确认以下架构决策:
> - CSS 变量不加命名空间前缀（独立部署，非嵌入式组件）
> - 前端通过 `GET /api/models` 获取模型数据（保留本地回退）
> - 单文件 HTML 开发 → 生产时拆分 CSS 到 `css/variables.css` + `css/style.css`
> - 上下文压缩策略已取消（现代 LLM 内置此能力）
> - 全局状态封装在 IIFE 中，仅通过 `window.__TC` 暴露公开 API

## 1. 模块概述

前端界面是 "Prompt 优化工作站" 的唯一用户触点。它采用**纯静态 HTML/CSS/JS** 架构，不依赖 React、Vue 或任何前端框架。所有数据通过 `fetch()` 异步获取，渲染完全由原生 DOM 操作完成。

**核心设计原则：**

- **零框架依赖** —— 单页应用，无构建步骤，浏览器打开 `index.html` 即可运行
- **状态集中** —— 所有 UI 状态收敛到单一 `AppState` 对象，避免分散管理
- **后端解耦** —— 不直接调用后端模块，全部请求经由 `api-gateway` 中转
- **离线可读** —— 页面布局和暗色主题在无网络时完整呈现，仅功能数据需等待 API

**与同类模块的关系：** 本模块是所有后端能力的展示层，不包含任何业务逻辑。压缩、分词、成本模拟均由后端模块完成，前端只负责发送请求和渲染结果。

## 2. 文件结构

```
frontend/
├── index.html           # 主入口：三面板 section 结构 + 顶部模型切换器
├── css/
│   ├── variables.css    # 全局 CSS 自定义属性（颜色、间距、字体、断点）
│   └── style.css        # 组件级样式（面板、卡片、按钮、表单、动效）
└── js/
    ├── app.js           # 应用内核：AppState 初始化、面板联动、API 调用、事件绑定
    └── config.js        # 可选覆盖文件：用户可按喜好覆盖默认配置（不提供时使用内置默认值）
```

### 2.1 各文件职责说明

| 文件 | 职责 | 关键约束 |
|------|------|----------|
| `index.html` | 定义三个面板的 DOM 骨架、顶部导航栏、全局模型切换器 HTML 结构 | 不包含内联 JS/CSS；所有资源通过 `<link>` 和 `<script>` 引入 |
| `variables.css` | 声明所有 CSS 自定义属性：色值、间距标尺、字体栈、圆角、阴影、z-index 层级 | 不包含业务样式；其他 CSS 文件一律引用此处的变量 |
| `style.css` | 面板布局网格、卡片样式、textarea/按钮/选择器等组件外观、响应式媒体查询 | 色值全部使用 `var(--xxx)` 引用 variables.css，无硬编码颜色 |
| `app.js` | `AppState` 状态对象、初始化流程编排（含 API 数据获取）、面板联动、事件绑定、HTTP 通信 | 直接持有 HTTP 通信逻辑（不另设 api.js 层）；通过 `window.__TC` 暴露公开 API |
| `config.js` | 可选覆盖文件，用户可在此文件中自定义默认配置（模型选择、偏好设置等） | 纯数据文件，不包含逻辑；缺失时不报错，应用使用内置默认值 |

## 4. 面板 1：原始输入

### 4.1 textarea 输入区

- 占据面板上部主要空间
- 占位文本："在此粘贴 Prompt 文本，或切换到多轮对话模式粘贴 JSON..."
- 支持直接输入和粘贴两种方式
- 高度自适应（随内容增长至面板最大高度，不溢出面板边界）
- 输入后实时触发字符计数和 token 预估

### 4.2 字符计数与预估 Token

- **字符数**：实时显示已输入字符总数，含空格和换行
- **预估 Token**（实时预览）：前端本地使用 `len(text) * 0.25` 快速估算，毫秒级响应，无网络开销
- **精确 Token 数**（压缩后）：压缩完成后调用 `POST /api/tokenize` 获取真实分词器的精确计数，展示在面板 3 的对比表中
- 当字符数超过上下文窗口时显示警告标识（黄色警示）
- **双重机制说明**：前端本地估算是为了实时响应（用户打字时即时看到参考值），后端真实分词器计数用于最终的精确对比。两者并存，互不替代。

→ 参见 [tokenizer-layer](../tokenizer-layer/README.md) 了解精确分词实现。自 Phase 1 起后端使用真实分词器（Tiktoken / Hf / SentencePiece / Mistral），替代 `len*0.25` 估算法。

### 4.3 输入模式切换

在 textarea 下方提供下拉切换器，两种模式：

| 模式 | 说明 | textarea 提示 |
|------|------|---------------|
| 自由文本 | 普通 Prompt 文本输入 | "在此粘贴 Prompt 文本..." |
| 多轮对话 | 粘贴 JSON 格式的多轮对话（messages 数组） | "粘贴 messages JSON..." |

切换模式时不丢失已有内容（用户可自由来回切换），但预估 token 的计算方式会根据模式自动调整：
- **自由文本**：按纯文本预估
- **多轮对话**：解析 JSON 后按对话结构（含角色标记、消息间隔）预估

## 5. 面板 2：压缩策略

### 5.1 策略选择器

两个单选项（radio group），互斥选择：

| 策略 | 标识符 | 适用场景 |
|------|--------|----------|
| 规则引擎 | `rule` | 快速压缩，确定性规则，适合结构化和重复性 Prompt |
| LLM 压缩 | `llm` | 语义级压缩，保留意图，适合复杂指令 |

每个选项旁附简短说明文字，帮助用户理解差异。

### 5.2 强度滑块

- 范围：`low` / `medium` / `high` 三档
- 视觉为连续滑块，吸附到三档位置（带刻度标记）
- 每档下方显示简短描述（如 "激进压缩，可能丢失细节""平衡压缩，推荐""轻度压缩，保留完整语义"）
- 强度影响最终压缩率，具体行为由 `compression-engine` 决定

→ 参见 [compression-engine](../compression-engine/README.md#强度参数)

### 5.3 LLM 配置面板（条件显示）

仅在策略选择 **LLM 压缩** 时展开，包含：

| 控件 | 类型 | 说明 |
|------|------|------|
| 模型选择 | 下拉框 | 使用哪组 LLM 执行压缩（列表来自 model-registry 中支持压缩的模型） |
| Temperature | 滑块 | 0.0 - 1.0，影响压缩时的创造力 |
| Max Tokens | 数字输入 | 限制压缩输出最大长度 |

→ 参见 [model-registry](../model-registry/README.md#模型能力分类)

### 5.4 操作按钮

| 按钮 | 触发动作 | 状态 |
|------|----------|------|
| **一键压缩** | 调用 `POST /api/compress`，传入策略、强度、文本 | 面板 2 核心 CTA，loading 时禁用 |
| 导出 | 将压缩结果导出为文本文件 | 仅在压缩完成后可用 |
| 撤销 | 回退到上一次压缩前的结果 | 仅在至少一次压缩后可用 |

- 月省金额实时计算（需先运行成本模拟器，否则显示灰色占位 "运行成本模拟以查看"）

### 6.2 多模型对比表

以表格形式展示同一输入经过不同模型/策略压缩后的结果对比：

| 模型 | 压缩后 Token | 节省 | 质量评分 | 操作 |
|------|-------------|------|----------|------|
| GPT-4o | 180 | 65% | A- | [预览] |

- **预览**：点击后在本面板弹出一个叠加层（overlay），展示该模型的压缩结果全文

**输出展示：**

对当前输入应用所有可用策略，按压缩率从高到低排序展示：

```
┌── 压缩率排行榜 ────────────────────────┐

│  1  规则引擎 (激进)     65%    0.3s    │
```

- 每行右侧显示耗时，帮助用户在压缩率和速度之间权衡
- 排行榜数据来自 `POST /api/compress/rank` 端点

在核心对比卡片下方，提供文本级别的原文 vs 压缩后对比展示：

```
┌──────────────────────────────────────────────────┐
│  ▣ 原文 vs 压缩后                                 │
│                                                  │
│  ┌─────────────────┐  ┌─────────────────────────┐ │
│  │  原文 (520 字)   │  │  压缩后 (180 字)         │ │
│  │                 │  │                         │ │
│  │  请帮我分析这份  │  │  分析数据，找出异常和趋势  │ │
│  │  数据，找出其中  │  │                         │ │
│  │  的异常和趋势... │  │  ~~请帮我~~  ~~这份~~    │ │
│  │                 │  │  ++新增: 可视化展示++     │ │
│  └─────────────────┘  └─────────────────────────┘ │

└──────────────────────────────────────────────────┘
```

**功能要点**：

| 特性 | 说明 |
|------|------|
| **并排对比** | 原文与压缩后文本左右或上下并排，保留原始换行和段落结构 |
| **差异高亮** | 删除内容使用红色 + ~~删除线~~，新增/改写内容使用绿色高亮 |

| **复制压缩版** | 点击压缩侧文本可选中全部，一键复制 |

**数据来源**：`POST /api/compress` 响应的 `original_text` 和 `compressed_text` 字段，前端使用 JS 实现差异对比算法（基于行级 LCS diff）。不依赖于后端进行 diff 计算。

## 7. 全局模型切换器

### 7.1 位置与外观

- 位于页面顶部导航栏右侧
- 下拉选择框，选项列表由 `model-registry` 提供
- 当前选中模型以品牌色高亮标记

### 7.2 联动逻辑

切换模型触发以下更新流程：

```
用户切换模型 → AppState.selectedModels 更新
                │
                ├─→ 面板1: 重新估算 token（因不同模型 tokenizer 可能不同）
                ├─→ 面板2: LLM 配置面板的默认模型更新（如当前策略为 LLM 压缩）
                ├─→ 面板3: 多模型对比表重新请求（如已有数据）
                └─→ 成本模拟器: 单价依据新模型更新
```

**关键约束：** 切换模型时不自动清除面板 1 已输入的文本，但所有依赖模型的结果区域显示 "数据已过时，请重新压缩" 提示。

### 7.3 多模型选择

支持同时选中多个模型（复选框式下拉，或标签式选择器）。当选多个模型时：
- 面板 3 的多模型对比表同时显示所有选中模型的结果
- 排行榜只显示选中模型的策略选项
- 成本模拟器分别计算每个模型的费用并排展示

→ 参见 [model-registry](../model-registry/README.md#模型清单)

## 9. 状态管理

### 9.1 AppState 对象

所有应用状态收敛到单一全局对象 `AppState`，定义在 `app.js` 中。

```javascript
const AppState = {
    // 全局模型
    selectedModels: ["GPT-4o"],      // 当前选中的模型列表
    allModels: [],                   // 来自 model-registry 的完整模型列表

    // 面板1：原始输入
    inputText: "",                   // textarea 中的文本
    inputMode: "text",               // "text" | "conversation"
    charCount: 0,                    // 字符计数
    estimatedTokens: 0,              // 预估 Token 数

    // 面板2：压缩策略
    compressionStrategy: "rule",     // "rule" | "llm"
    compressionLevel: "medium",      // "low" | "medium" | "high"
    llmConfig: {                     // LLM 压缩专属配置
        model: "",
        temperature: 0.3,
        maxTokens: 1024
    },

    // 面板3：结果
    compressedText: "",              // 压缩结果文本
    compressionStats: null,          // { original, compressed, saved, savedPercent }
    tokenResults: null,              // 各模型 token 对比结果
    costSimulation: null,            // 成本模拟结果
    rankResults: [],                 // 排行榜

    // 系统
    isLoading: false,                // 是否正在请求
    lastError: null,                 // 上一次错误对象
    lastCompressedAt: null,          // 时间戳
    history: []                      // 压缩历史栈（用于撤销）
};
```

### 9.2 数据流

```
┌─────────────────────────────────────────────────────────────────┐
│                        用户操作触发                               │
│                                                                  │
│  输入文本 ──→ 实时更新 inputText, charCount                      │
│                │                                                 │
│                └─→ 前端本地估算 len(text) * 0.25                 │
│                        ↓  （毫秒级，无网络开销）                  │
│                   更新 estimatedTokens（实时预览值）              │
│                                                                  │
│  点击一键压缩 ──→ 更新 isLoading = true                          │
│                   保存当前快照到 history                          │
│                   │                                               │
│                   ├─→ 调用 api-gateway POST /compress            │
│                   │       ↓                                      │
│                   │   更新 compressedText                        │
│                   │   更新 compressionStats                      │
│                   │                                               │
│                   └─→ 调用 api-gateway POST /tokenize            │
│                           ↓  （使用真实分词器精确计数）            │
│                       更新 tokenResults（精确值，覆盖本地估算）    │
│                       更新 isLoading = false                     │
│                                                                  │
│  加载对比表 ──→ 复用 tokenResults 数据（由 compress 流程填充）    │
│                                                                  │
│  更改月调用量 ──→ 调用 api-gateway POST /cost/simulate           │
│                     ↓                                            │
│                 更新 costSimulation                              │
└─────────────────────────────────────────────────────────────────┘
```

**数据流要点**：
- 输入阶段使用前端本地估算（`len*0.25`）保证实时响应，无需等待 API
- 压缩完成后调用 `POST /api/tokenize` 获取真实分词器（Tiktoken/Hf/SentencePiece/Mistral）的精确 token 数
- 精确值覆盖本地估算值，展示在面板 3 的对比表中
- `POST /api/compress/rank` 和 `POST /api/compress/compare` 端点已合并到统一压缩流程中——对比表和排行榜由前端在压缩完成后对 tokenResults 做排序和计算得出

### 9.3 状态变更通知

`app.js` 实现一个简易的观察者模式（Observable）：

- `AppState.onChange(key, callback)` —— 注册状态变更回调
- `AppState.emit(key, newValue)` —— 触发所有已注册的对应回调

各面板组件通过 `onChange` 订阅自己关心的状态字段，当字段变化时自动重渲染。例如面板 3 订阅 `compressionStats`，当该字段更新时自动刷新核心对比卡片的数字。

### 9.4 撤销机制

- 每次点击 "一键压缩" 前，将当前 `AppState` 中与结果相关的字段（`compressedText`、`compressionStats`）快照保存到 `history` 栈
- 点击 "撤销" 时从 `history` 弹出上一次快照并恢复
- 栈最大深度 10 层，超出时移除最早记录
- 撤销不重发 HTTP 请求（数据从内存恢复）

### 9.5 初始化流程

应用加载时，按以下顺序获取模型数据：

```
页面加载 → app.js 初始化
            │
            ├─→ 检查 sessionStorage 是否有缓存的模型数据
            │     ├─ 有 → 直接使用，跳过 API 请求
            │     └─ 无 → 发起 GET /api/models
            │              ├─ 成功 → 缓存到 sessionStorage，更新 AppState.allModels
            │              └─ 失败 → 使用硬编码的最小化模型列表（仅 o200k_base）
            │
            └─→ AppState 就绪 → 触发各面板初始渲染
                 各面板通过 AppState.onChange 订阅需要的字段
```

**关键约束**:
- 模型数据仅缓存到 `sessionStorage`（会话级），页面关闭即清除
- 硬编码回退列表只包含 1-2 个最常用模型（如 GPT-4o），足够展示界面布局
- 初始化是异步的：页面骨架屏先渲染，模型数据加载完成后各面板再填充内容
- API 超时设为 5 秒，超时后直接使用回退列表

## 12. API 交互层

### 12.1 HTTP 通信模块（`app.js` 内联）

HTTP 通信逻辑封装在 `app.js` 内部的 `ApiClient` 对象中，不另设独立的 `api.js` 文件。

```javascript
const ApiClient = {
    baseURL: "/api",
    timeout: 10000,             // 默认超时 10s
    abortController: null,       // 用于取消上次未完成的请求

    async get(path, params) { /* ... */ },
    async post(path, body) { /* ... */ }
};
```

### 12.2 请求方法

| 方法 | 参数 | 返回 | 说明 |
|------|------|------|------|
| `get(path, params?)` | `params` 自动拼接到 URL query string | 解析后的 JSON 对象 | 仅用于只读数据请求 |
| `post(path, body)` | `body` 序列化为 JSON | 解析后的 JSON 对象 | 用于所有写入/计算请求 |

### 12.3 统一错误处理

所有 API 调用经过以下错误处理流水线：

```
请求发送
    │
    ├─→ 网络错误（fetch 失败 / DNS 问题）
    │       └─→ 返回 { error: true, code: "NETWORK_ERROR", message: "..." }
    │
    ├─→ 超时（AbortController 触发）
    │       └─→ 返回 { error: true, code: "TIMEOUT", message: "请求超时" }
    │
    ├─→ HTTP 4xx（客户端错误）
    │       └─→ 透传后端 error 对象，附加 status code
    │
    ├─→ HTTP 5xx（服务端错误）
    │       └─→ 返回 { error: true, code: "SERVER_ERROR", message: "服务暂时不可用" }
    │
    └─→ 解析失败（响应非 JSON）
            └─→ 返回 { error: true, code: "PARSE_ERROR", message: "无法解析响应" }
```

### 12.4 取消机制

- 每次发起新请求前，先取消上一次同一端点的未完成请求
- `ApiClient` 内部维护 `abortController`，新请求创建新的 `AbortController`，调用上一次的 `.abort()`
- 被取消的请求静默吞掉 `AbortError`，不触发错误显示

### 12.5 与 app.js 的对接

`app.js` 不直接调用 `fetch()`，所有数据获取通过以下规约：

```
app.js 事件处理器
    │
    ├─→ 更新 AppState（设置 isLoading = true）
    │
    ├─→ 调用 ApiClient.post(path, body)
    │       │
    │       └─→ 返回后，app.js 处理结果
    │               │
    │               ├─→ 成功: 更新 AppState 对应字段，触发 UI 渲染
    │               └─→ 失败: 更新 AppState.lastError，触发 toast 显示
    │
    └─→ 更新 AppState（设置 isLoading = false）
```

### 12.6 数据转换：snake_case → camelCase

后端 API 返回的字段使用 snake_case（如 `group_id`、`model_name`），前端内部使用 camelCase（如 `groupId`、`modelName`）。`app.js` 在收到 API 响应后立即执行转换：

```
GET /api/models 响应
    │  { "groups": [ { "group_id": "o200k_base", "display_name": "...", ... } ] }
    ▼
app.js 中的 apiGroupToModelGroup(groups) 转换器
    │  - 将每个字段从 snake_case 映射为 camelCase
    │  - 嵌入 pricing 数据到 models 数组的每个元素中
    │  - 附加 computed 字段（如 isAvailable, displayLabel）
    ▼
AppState.allModels
    │  [ { groupId: "o200k_base", displayName: "...", models: [...], ... } ]
    ▼
各面板渲染函数消费 AppState 中的 camelCase 数据
```

**转换器签名**: `function apiGroupToModelGroup(apiGroups: Array): Array`

**关键约束**:
- 转换在 API 响应后一次性完成，不反复转换
- `sessionStorage` 中缓存的是原始 snake_case 数据（节省存储空间），每次从缓存读取时重新转换
- 转换器只做字段名映射和结构重组，不做数据校验——校验由 Python 后端 Pydantic 层完成

## 13. 与其他模块的关系

### 13.1 依赖关系图

```
┌──────────────┐     HTTP      ┌────────────────┐
│  frontend-ui │ ──────────→   │  api-gateway    │
│  (本模块)     │ ←──────────   │  (唯一入口)      │
└──────────────┘     JSON      └───────┬────────┘
                                       │
                         ┌─────────────┼─────────────┐
                         │             │             │
                         ▼             ▼             ▼
                   ┌──────────┐ ┌──────────┐ ┌────────────┐
                   │compression│ │  cost-   │ │ tokenizer  │
                   │-engine   │ │ simulator│ │ -layer     │
                   └──────────┘ └──────────┘ └────────────┘
                         │
                         ▼
                   ┌──────────────┐
                   │model-registry│
                   └──────────────┘
```

### 13.2 详细依赖

**运行时依赖 [api-gateway](../api-gateway/README.md)**
- 所有数据通过 HTTP 请求经过 api-gateway，前端不直接访问任何后端模块
- api-gateway 负责路由分发、鉴权（如有）、请求日志
- 前端仅感知 api-gateway 暴露的端点路径，不感知后端内部拓扑

**使用 [model-registry](../model-registry/README.md) 的数据**
- 全局模型切换器的选项列表来自 `GET /api/models`
- LLM 压缩配置面板中的模型下拉框也来自同一列表，但只筛选 `capabilities.compression = true` 的模型
- 模型单价信息用于成本模拟器计算

**与 [compression-engine](../compression-engine/README.md) 配合**
- 面板 2 的 "一键压缩" 按钮触发 `POST /api/compress`，最终由 compression-engine 执行
- 策略选择（rule / llm / context）和强度参数（low / medium / high）映射到 engine 的压缩策略配置
- 多模型对比和排行榜最终也基于 compression-engine 的能力

**与 [cost-simulator](../cost-simulator/README.md) 配合**
- 面板 3 的成本模拟器输入（月调用量、Token 比例）发送至 `POST /api/cost/simulate`
- 成本数据在后端由 cost-simulator 计算，前端仅渲染结果图表

### 13.3 无直接依赖的模块

前端界面与以下模块无直接接口关系，业务逻辑完全通过 api-gateway 封装：

- `tokenizer-layer` —— 前端仅通过 `POST /api/tokenize` 调用，不直接引用分词器
- 任何数据库/存储层模块
- 任何认证/鉴权模块（如集成，由 api-gateway 统一处理）

## 14. 扩展指南

### 14.1 添加新面板区域

如需要在现有三面板之外增加第四面板或子区域，按以下步骤操作：

1. **HTML 框架**：在 `index.html` 的目标面板 `<section>` 内添加新的 `<div class="card">` 区域
2. **状态字段**：在 `app.js` 的 `AppState` 中添加对应状态字段
3. **渲染函数**：在对应的 JS 文件中（或 app.js 的渲染部分）添加 `renderXxx()` 函数
4. **状态订阅**：调用 `AppState.onChange(key, this.renderXxx)` 注册更新回调
5. **API 端点**：如需要后端数据，在 `app.js` 的 `ApiClient` 中添加方法（若通用则复用现有方法）
6. **样式**：在 `style.css` 中编写组件样式，引用 `var(--xxx)` 变量
7. **响应式**：在媒体查询中调整该区域在不同断点下的布局

**示例**：添加 "压缩历史" 区域
- `AppState.history` 已存在，直接复用
- `renderHistory()` 读取 `history` 渲染表格
- 订阅 `AppState.onChange("history", renderHistory)`

### 14.2 添加新图表类型

当前仅支持纯 CSS 柱状图（通过 `app.js` 中的 `renderChart()` 系列函数生成带内联样式的 DOM 元素）。如需新图表类型：

1. **在 `app.js` 中添加工厂函数**：
   - 接受数据对象 `{ labels: [], values: [], colors: [] }`
   - 返回 HTML 字符串或 `DocumentFragment`
   - 保持纯函数风格，不访问 `AppState`

2. **图表类型矩阵**（当前支持与规划）：

| 类型 | 状态 | 实现方式 |
|------|------|----------|
| 水平柱状图 | 已支持 | `renderHorizontalBar(data)` |
| 垂直柱状图 | 已支持 | `renderVerticalBar(data)` |
| 进度条 | 已支持 | `renderProgressBar(percent, color)` |
| 折线图 | 规划中 | 可引入轻量 SVG 或 Canvas |
| 饼图 | 规划中 | 使用 CSS conic-gradient 模拟 |

3. **注册新图表**：在 `app.js` 的渲染章节中添加新图表函数，将渲染结果插入目标 DOM 容器。
4. **暗色兼容**：确保颜色使用 `var(--xxx)` 或在 JS 中读取计算样式以适配主题。
5. **动画**：如需要入场动画，在图表函数中使用 `requestAnimationFrame` 或 CSS `@keyframes`。

### 14.3 添加新策略选项

如需在面板 2 新增压缩策略（如 "字典压缩"）：

1. 在 `index.html` 的策略选择器组中添加新的 radio 选项
2. 确定是否需要新的配置面板（如需要，仿照 LLM 配置面板的条件显示模式）
3. `AppState` 中的 `compressionStrategy` 枚举值扩展到新值
4. 调用 `POST /api/compress` 时传入新策略名称（后端需同步支持）
5. 排行榜将自动包含新策略（由后端 `POST /api/compress/rank` 返回全量结果）

### 14.4 扩展建议

- **避免在 `index.html` 中添加内联 `<style>` 或 `<script>`** —— 所有样式和逻辑必须放在对应文件中
- **新功能优先作为面板 3 的子区域** —— 面板 3 宽度为 50%，视觉空间最大，最适合展示结果型内容
- **图表库选择**：如纯 CSS 图表不满足需求，考虑引入 `Chart.js`（轻量，40KB gzip）或纯 SVG 手写，不建议引入重量级库
- **性能注意**：`AppState.history` 中每条快照包含完整的文本内容（可能很长），注意控制最大深度
