# Token Calculator 项目复盘

> 2026-07-14

---

## 项目定位

一个在线 Prompt 压缩服务网站。用户输入提示词，系统压缩后返回精简版本，节省 Token 消耗以实现省钱。

## 最终结果

**失败。** 三个核心目标均未达成：

1. **前端不符合设计预期** — 最终界面与最初设想相去甚远
2. **规则压缩能力不足** — 程序性压缩对结构化 Prompt 效果有限
3. **LLM 压缩得不偿失** — 调用大模型压缩消耗的 Token 比节省的还多，商业逻辑不成立

---

## 失败原因分析

### 一、Agent 上下文退化

即使有 1M 上下文窗口，Agent 在复杂项目中仍会出现明显的能力下滑。

项目初期表现很好：讨论设计、总结路线、规划结构、为每个模块编写独立文档，这些都很顺利。但进入实施阶段后，随着项目文件增多、上下文膨胀，Agent 的决策质量持续下降。到项目中后期，Agent 开始做出明显不符合最初设计的决策，而这些偏差未得到及时纠正。

### 二、Agent 隔离缺失

没有为不同阶段的 Agent 做好边界隔离。实施阶段的 Agent 不是从单个模块文档出发、逐个实现模块，而是从总项目文档出发，一口气实现整个层面的设计。

正确的做法应该是：每个模块的 Agent 只拿到该模块的接口文档和约束条件，按模块逐个交付和验证。而非让一个 Agent 看到全局后试图一次性覆盖所有东西。

### 三、Subagent 滥用

调用了数百个 Subagent 进行测试，消耗了数亿 Token。这些 Subagent 各自发现问题、解决问题，但彼此之间没有信息共享——同一个问题在多个 Subagent 中被反复发现、反复修复，造成巨额 Token 浪费。

### 四、接口约束不完整

虽然在总项目文档中编写了所有模块的接口设计，但 Subagent 使用的文档未做完整约束。导致：
- 接口不一致 — 不同模块的输入输出格式对不上
- 技术栈冲突 — 不同 Agent 选择了不同的实现方式
- 最终拼接时发现无法对接

### 五、缺乏工作跟踪

项目的实际进展完全依赖于 Agent 的口头转述，没有清晰的跟踪文档。

- 不知道哪些模块已完成、哪些在进行中、哪些卡住了
- 不知道每个模块的实际状态与计划的偏差
- 直到项目"完成"后才发现大量问题

### 六、缺乏工作记录

已解决的问题没有记录。导致：
- 同一问题在不同 Subagent 中重复出现
- 每次都要走完整的"发现问题 → 排查原因 → 修复 → 验证"流程
- 无法从已解决的问题中提取经验

### 七、零技术积累

项目做完后，没有任何技术积累留存。所有的经验、踩过的坑、解决的难题，都随着 Agent 上下文消失。现在只能靠人脑回忆并手动转述这个过程，这势必出现遗漏。

---

## 教训总结

| # | 问题 | 应如何做 |
|---|------|---------|
| 1 | Agent 上下文退化 | 复杂项目拆分为多个独立 session，每个 session 只负责一个模块 |
| 2 | Agent 隔离缺失 | 每个 Agent 只给当前模块文档，不给全局视图 |
| 3 | Subagent 滥用 | Subagent 必须有明确边界，结果需汇总去重后再进入下一步 |
| 4 | 接口约束不完整 | 接口文档要有机器可校验的格式（如 Pydantic model），Subagent 必须遵循 |
| 5 | 缺乏工作跟踪 | 建立类似 `问题跟踪/` 的任务跟踪系统，Agent 完成一个更新一个 |
| 6 | 缺乏工作记录 | 每个决策、每个已解决 Bug 都需记录，不可依赖 Agent 记忆 |
| 7 | 零技术积累 | 项目结束后必须产出复盘文档 + 可复用的知识条目 |

## 与 skills/问题跟踪 的关系

`/home/hcj/dev/school-system/问题跟踪/` 中的方法论——状态流转、编号索引、每个问题独立文件夹、截图和文档同行——本质上就是一套**人工驱动的工作跟踪系统**。如果 Token Calculator 项目在实施阶段有类似的跟踪机制，至少第 5、6、7 条问题不会这么严重。
